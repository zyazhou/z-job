# JZero 应用场景案例 — 通用任务调度场景

> 版本：1.0  
> 日期：2026-03-21  
> 说明：JZero 是通用型任务调度框架，适用于各类业务场景  

---

## 一、框架通用性说明

### 1.1 JZero 的定位

```
┌─────────────────────────────────────────────────────────────────────┐
│                        JZero 核心定位                                │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│   JZero = 通用分布式任务调度框架                                     │
│                                                                      │
│   ├── 不限行业：电商、金融、IoT、游戏、社交、企业服务...             │
│   ├── 不限场景：定时任务、延时任务、事件触发、批处理...             │
│   ├── 不限语言：Java、Go、Python、Node.js 多语言执行器              │
│   └── 不限规模：单机开发、小团队、中大型企业级部署                   │
│                                                                      │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.2 框架能力矩阵

| 能力维度 | 支持情况 | 说明 |
|----------|----------|------|
| **调度类型** | ✅ 全面支持 | CRON、固定频率、固定延迟、一次性、事件触发 |
| **路由策略** | ✅ 8 种 | 轮询、随机、一致性哈希、故障转移、分片广播等 |
| **执行器** | ✅ 多语言 | Java(gRPC)、Go(gRPC)、Python(gRPC)、Node.js |
| **可靠性** | ✅ 生产级 | 失败重试、超时控制、阻塞策略、任务幂等 |
| **集群** | ✅ 高可用 | 主从选举、故障自动切换、WAL 状态同步 |
| **网关** | ✅ 轻量 | 路由、负载均衡、鉴权、限流、熔断 |

## 二、通用场景示例

### 场景一：定时对账 — 金融行业

```
业务场景：
┌─────────────────────────────────────────────────────────────────────┐
│  每天凌晨 2:00，对昨日交易进行银行对账                               │
│                                                                      │
│  需求：                                                             │
│  ├── 读取昨日所有交易记录                                            │
│  ├── 调用银行接口获取对账单                                          │
│  ├── 比对差异，生成差错报表                                          │
│  ├── 发送邮件给财务人员                                              │
│  └── 对差异交易进行自动调账                                          │
│                                                                      │
│  JZero 方案：                                                       │
│  ├── Cron: 0 0 2 * * ? (每天凌晨 2 点)                           │
│  ├── 分片广播：将交易数据分 10 个分片，并行处理                     │
│  ├── 失败重试：单笔失败不影响整体，支持重跑                          │
│  └── 执行日志：实时查看处理进度，定位问题                            │
└─────────────────────────────────────────────────────────────────────┘

技术实现：
┌─────────────────────────────────────────────────────────────────────┐
│  @JZeroJob("dailyReconciliation")                                   │
│  public ReturnT<String> reconcile(JobContext ctx) {                 │
│      int shardIndex = ctx.getShardIndex();                         │
│      int shardTotal = ctx.getShardTotal();                         │
│                                                                      │
│      // 分片处理：每个执行器处理 1/10 的数据                        │
│      List<Transaction> txs = transactionService                    │
│          .queryByDate(shardIndex, shardTotal, yesterday());         │
│                                                                      │
│      for (Transaction tx : txs) {                                   │
│          ReconcileResult result = bankClient.reconcile(tx);         │
│          if (!result.isMatch()) {                                   │
│              errorService.save(result);                            │
│          }                                                          │
│      }                                                              │
│      return ReturnT.SUCCESS;                                        │
│  }                                                                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 场景二：数据同步 — 物联网

```
业务场景：
┌─────────────────────────────────────────────────────────────────────┐
│  每分钟同步 100 万+ 物联网设备的状态数据                            │
│                                                                      │
│  需求：                                                             │
│  ├── 设备数量：100 万+                                              │
│  ├── 采集频率：每分钟                                              │
│  ├── 响应要求：5 分钟内完成                                         │
│  └── 可靠性：数据不能丢失                                           │
│                                                                      │
│  JZero 方案：                                                       │
│  ├── 固定频率：每分钟触发                                           │
│  ├── 分片广播：100 个执行器，每个处理 1 万台设备                    │
│  ├── 批量处理：每批 1000 条，批量写入                              │
│  └── 故障转移：执行器宕机，任务自动转移到其他节点                   │
└─────────────────────────────────────────────────────────────────────┘

技术实现：
┌─────────────────────────────────────────────────────────────────────┐
│  @JZeroJob("deviceDataSync")                                        │
│  public ReturnT<String> sync(JobContext ctx) {                      │
│      int shardIndex = ctx.getShardIndex();                         │
│      int shardTotal = ctx.getShardTotal();                         │
│                                                                      │
│      // 分片获取设备数据                                            │
│      List<DeviceData> devices = deviceService                       │
│          .getDevicesByShard(shardIndex, shardTotal);                │
│                                                                      │
│      // 批量同步到时序数据库                                         │
│      for (List<DeviceData> batch : Lists.partition(devices, 1000)) │
│          tsdb.batchWrite(batch);                                    │
│                                                                      │
│      return ReturnT.SUCCESS.put("count", devices.size());            │
│  }                                                                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 场景三：消息推送 — 社交/运营

```
业务场景：
┌─────────────────────────────────────────────────────────────────────┐
│  运营活动消息推送，需要准实时触达 1000 万用户                        │
│                                                                      │
│  需求：                                                             │
│  ├── 用户规模：1000 万                                              │
│  ├── 推送速度：越快越好                                             │
│  ├── 到达率：> 95%                                                 │
│  └── 避免骚扰：每个用户每天最多接收 3 条                            │
│                                                                      │
│  JZero 方案：                                                       │
│  ├── 事件触发：运营活动发布时立即触发                               │
│  ├── 分片广播：1000 万用户分 100 个分片，并行推送                   │
│  ├── 限流保护：每秒推送量不超过 10 万                               │
│  └── 失败重试：推送失败自动重试 3 次                                │
└─────────────────────────────────────────────────────────────────────┘

技术实现：
┌─────────────────────────────────────────────────────────────────────┐
│  @JZeroJob("pushNotification")                                      │
│  public ReturnT<String> push(JobContext ctx) {                      │
│      String activityId = ctx.getParam();                           │
│                                                                      │
│      // 分片推送：每个执行器负责一部分用户                          │
│      int shardIndex = ctx.getShardIndex();                         │
│      int shardTotal = ctx.getShardTotal();                         │
│                                                                      │
│      List<Long> userIds = userService                              │
│          .getUserIdsByShard(shardIndex, shardTotal, activityId);   │
│                                                                      │
│      int successCount = 0;                                         │
│      for (Long userId : userIds) {                                 │
│          try {                                                       │
│              pushService.push(userId, activityId);                 │
│              successCount++;                                        │
│          } catch (Exception e) {                                   │
│              // 失败不影响其他用户                                  │
│              log.error("推送失败 userId={}", userId, e);           │
│          }                                                          │
│      }                                                              │
│      return ReturnT.SUCCESS.put("success", successCount);           │
│  }                                                                   │
└─────────────────────────────────────────────────────────────────────┘
```

### 场景五：订单处理 — 电商/零售

```
业务场景：
┌─────────────────────────────────────────────────────────────────────┐
│  订单创建 30 分钟后未支付，自动取消并释放库存                         │
│                                                                      │
│  需求：                                                             │
│  ├── 订单量：日均 100 万                                            │
│  ├── 处理延迟：< 5 分钟                                             │
│  └── 准确性：不能误取消                                            │
│                                                                      │
│  JZero 方案：                                                       │
│  ├── 固定延迟：订单创建后 30 分钟触发                               │
│  ├── 分片处理：按订单 ID 分片，并行处理                             │
│  ├── 幂等设计：防止重复取消                                         │
│  └── 异步执行：不阻塞下单流程                                       │
└─────────────────────────────────────────────────────────────────────┘

技术实现：
┌─────────────────────────────────────────────────────────────────────┐
│  @JZeroJob("cancelUnpaidOrder")                                     │
│  public ReturnT<String> cancel(JobContext ctx) {                    │
│      String batchId = ctx.getParam();                              │
│                                                                      │
│      // 查询待取消订单（30 分钟前创建，未支付）                     │
│      List<Order> orders = orderService                              │
│          .queryUnpaidOrders(batchId, 30 * 60);                    │
│                                                                      │
│      int cancelCount = 0;                                          │
│      for (Order order : orders) {                                   │
│          // 幂等检查                                                │
│          if (!idempotentService.tryLock(order.getId()))            │
│              continue;                                             │
│                                                                      │
│          try {                                                       │
│              orderService.cancel(order.getId());                   │
│              inventoryService.releaseStock(order.getItems());      │
│              cancelCount++;                                         │
│          } catch (Exception e) {                                   │
│              log.error("取消订单失败 {}", order.getId(), e);       │
│          }                                                          │
│      }                                                              │
│      return ReturnT.SUCCESS.put("cancelCount", cancelCount);        │
│  }                                                                   │
└─────────────────────────────────────────────────────────────────────┘
```

## 四、场景案例：通用任务调度平台

### 4.1 案例：企业级任务调度平台

```
┌─────────────────────────────────────────────────────────────────────┐
│               某中大型企业通用任务调度平台                            │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
│  背景：                                                             │
│  ├── 业务线：5 条业务线，每条业务线有多个定时任务                    │
│  ├── 任务数量：200+ 定时任务                                        │
│  ├── 团队构成：Java、Go、Python 三个开发团队                       │
│  └── 痛点：                                                         │
│      ├── 任务分散在各个系统中，缺少统一管理                          │
│      ├── 部分任务用 XXL-JOB，部分用 Quartz，运维困难                 │
│      ├── Go 和 Python 团队无法接入现有调度系统                       │
│      └── 故障排查困难，不知道任务执行情况                            │
│                                                                      │
│  解决方案：                                                         │
│  ├── 部署 JZero 集群（3 节点）                                     │
│  ├── Java 团队使用 Spring Boot Starter 接入                         │
│  ├── Go 团队使用 gRPC 执行器                                        │
│  ├── Python 团队使用 gRPC 执行器                                    │
│  └── 统一管理所有定时任务                                           │
│                                                                      │
│  效果：                                                             │
│  ├── 统一管理：所有任务在 JZero 控制台统一管理                       │
│  ├── 多语言支持：三个团队都能接入                                   │
│  ├── 可观测：实时查看任务执行日志，失败立刻告警                      │
│  └── 高可用：故障自动切换，0 人工干预                               │
└─────────────────────────────────────────────────────────────────────┘
```

### 6.2 选型建议

```
┌─────────────────────────────────────────────────────────────────────┐
│                         何时选择 JZero                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                      │
```
