# JZero Framework V1.0 — 产品文档

> 版本：1.0  
> 日期：2026-03-21  
> 定位：面向 Java 生态的**分布式任务调度 + 微服务治理**一体化框架  
> 核心特性：分布式集群 · 主从高可用 · 高吞吐 · 高并发 · 低延迟

---

## 一、产品背景

### 痛点

| 现有方案 | 问题 |
|----------|------|
| XXL-JOB | 调度中心单点风险，无原生主从选举；与微服务体系割裂 |
| Spring Cloud | 无内置任务调度；组件繁多，学习成本高 |
| go-zero | Go 生态，Java 团队无法直接使用 |
| Quartz 集群 | 基于数据库行锁，高并发下性能瓶颈明显 |

### 机会

Java 生态缺少一款**同时具备**：
1. 分布式任务调度（对标 XXL-JOB）
2. 微服务治理能力（对标 go-zero）
3. **原生集群 + 主从高可用 + 高性能**

的一体化框架。JZero 填补这一空白。

## 三、核心能力

### 3.1 分布式集群 & 主从高可用

这是 JZero 区别于 XXL-JOB 的最核心差异。

**主从选举**
- 基于 Raft 协议（或 Redis 轻量实现）自动选举 Master 节点
- Master 负责任务调度触发，Slave 实时同步状态热备
- Master 宕机后，Slave 在 **3 秒内**完成选举接管，任务不丢失

**节点角色**

```
┌─────────────────────────────────────────────────────┐
│                   JZero Cluster                      │
│                                                      │
│   ┌──────────────┐    ┌──────────────┐              │
│   │  Master Node │◄──►│  Slave Node  │              │
│   │  (主调度)     │    │  (热备)      │              │
│   └──────┬───────┘    └──────────────┘              │
│          │  故障自动切换                               │
│   ┌──────▼───────┐    ┌──────────────┐              │
│   │  Slave Node  │    │  Slave Node  │              │
│   │  (可扩展)     │    │  (可扩展)    │              │
│   └──────────────┘    └──────────────┘              │
└─────────────────────────────────────────────────────┘
```

**数据一致性**
- 任务状态变更通过 WAL（Write-Ahead Log）同步到所有 Slave
- 支持最终一致性模式（高性能）和强一致性模式（金融场景）

---

### 3.2 分布式任务调度

**调度方式**
- CRON 表达式（精确到秒）
- 固定频率 / 固定延迟
- 一次性任务（指定时间点触发）
- 事件驱动触发（监听 MQ 消息触发）

**执行器管理**
- 执行器**自动注册**，心跳保活，宕机自动摘除
- 支持执行器水平扩展，无上限
- 执行器分组管理（按业务线/环境隔离）

**路由策略（8种）**

| 策略 | 适用场景 |
|------|----------|
| 轮询 | 均匀分发，通用场景 |
| 随机 | 简单负载均衡 |
| 一致性哈希 | 相同参数路由同一节点，有状态任务 |
| 最不经常使用 | 避免热点节点 |
| 故障转移 | 主节点故障自动切换 |
| 分片广播 | 大数据量并行处理 |
| 忙碌转移 | 执行器繁忙时转移到空闲节点 |
| 权重轮询 | 按节点性能差异分配 |

**任务可靠性**
- 失败自动重试（可配置次数和间隔）
- 超时强制中断
- 阻塞处理：串行排队 / 丢弃后续 / 覆盖旧任务
- 任务幂等保障（防止集群重复触发）

**分片任务**
- 广播模式下自动分配 `shardIndex / shardTotal`
- 业务代码按分片处理数据，天然支持大数据量并行

---

### 3.3 微服务治理

**API 网关**
- 统一流量入口，动态路由
- 请求鉴权（JWT / API Key / OAuth2）
- 全局限流（令牌桶，支持集群共享计数）
- 请求/响应日志，链路追踪透传

**服务注册与发现**
- 内置轻量注册中心（开箱即用）
- 可对接 Nacos / Etcd / Consul（生产推荐）
- 服务健康检查（HTTP / TCP / gRPC 探针）
- 服务元数据：版本、分组、权重、标签

**负载均衡**
- 轮询、加权轮询、随机、最少连接数
- 灰度发布（按标签/版本路由）

**熔断与降级**
- 熔断器（基于错误率 + 响应时间双维度）
- 自动半开探测，故障恢复无需人工干预
- Fallback 降级函数，优雅返回兜底数据

**中间件链**
- 类 go-zero 风格，链式组合，零侵入
- 内置：日志、鉴权、限流、熔断、链路追踪、CORS
- 自定义中间件一行注解接入

### 3.4 高性能设计

| 能力 | 实现方式 | 效果 |
|------|----------|------|
| 异步非阻塞通信 | Netty 4.x NIO | 万级并发连接 |
| 时间轮调度 | HashedWheelTimer | 调度延迟 < 1ms |
| 无锁队列 | Disruptor | 百万级 TPS 任务入队 |
| 连接池复用 | 执行器长连接池 | 消除连接建立开销 |
| 批量触发 | 预读 + 批量下发 | 减少 DB 压力 80% |
| 零拷贝传输 | Netty FileRegion | 日志传输性能提升 3x |

### 3.5 可观测性

- **执行日志**：在线实时查看，支持关键字搜索、滚动加载
- **任务监控**：执行成功率、平均耗时、P99 延迟趋势图
- **集群状态**：节点角色、心跳状态、负载分布可视化
- **告警**：失败率超阈值 → 邮件 / Webhook / 钉钉通知
- **链路追踪**：兼容 OpenTelemetry，对接 Jaeger / Zipkin

## 五、部署模式

### 开发环境（单机）
```yaml
jzero:
  cluster:
    mode: standalone   # 单机模式，无需 Redis 选举
```

### 生产环境（集群）
```yaml
jzero:
  cluster:
    mode: cluster
    nodes:
      - host: 10.0.0.1  port: 7777   # Node 1
      - host: 10.0.0.2  port: 7777   # Node 2
      - host: 10.0.0.3  port: 7777   # Node 3
    election: redis     # redis / raft
```

### 推荐生产拓扑
- **JZero 集群**：3 节点（1 Master + 2 Slave），奇数节点保证选举
- **执行器**：按业务线水平扩展，无上限
- **数据库**：MySQL 主从，JZero 读写分离
- **缓存**：Redis Sentinel 或 Cluster
